## Context

项目当前存在教程与剧情触发逻辑，但缺少统一的系统对话框组件。现有文本提示与音频播放能力分散，导致在需要“底部对话框 + 逐字文本 + 句级音频”时需要重复拼装逻辑，且难以保证一致体验。

该改动将新增一个可复用的对话系统层：以 UI 组件负责视觉与交互，以控制器负责句子队列、打字机状态机和音频同步。目标是在不侵入现有玩法脚本的前提下，通过简洁 API 被教程、任务或剧情触发器调用。

## Goals / Non-Goals

**Goals:**
- 提供底部全宽黑色对话框，支持按需显示/隐藏，文本白色居中。
- 提供逐字展示能力，支持可配置字速与整句补全（skip）。
- 支持每句话绑定单独音频并在句子生命周期内正确播放/停止。
- 提供流程友好的调用接口（播放句子列表、监听句子完成与整段完成）。
- 保证在重复触发、场景切换、暂停恢复时行为可预期。

**Non-Goals:**
- 不实现分支对话树、选项选择、存档回放等复杂叙事系统。
- 不在本次改动中引入本地化工具链或字幕富文本排版系统。
- 不替换所有现有提示 UI，仅提供新系统对话能力并允许逐步接入。

## Decisions

1. **采用“数据 + 控制器 + 视图”分层**
   - 决策：定义句子数据结构（文本、音频、可选参数），由控制器管理播放状态，UI 视图只负责渲染。
   - 理由：便于在不同触发源复用，并减少 UI 脚本承担业务状态。
   - 备选：单脚本同时管理 UI 与流程。缺点是耦合高、难测试、后续扩展成本高。

2. **逐字效果使用协程驱动**
   - 决策：使用协程按字推进，统一处理字速、跳过、句子完成回调。
   - 理由：与 Unity 主线程 UI 更新模型匹配，易于控制暂停与中断。
   - 备选：`Update` 手写状态机。缺点是代码分散、可读性和维护性较差。

3. **句级音频与句子状态绑定**
   - 决策：每句开始时播放对应 `AudioClip`，句子结束或切换时停止/切换当前音频。
   - 理由：保证“所见即所听”的同步关系，避免音频跨句残留。
   - 备选：整段统一播放一个音频轨。缺点是无法精确匹配逐句展示需求。

4. **对外接口以“播放会话”为单位**
   - 决策：提供 `PlayDialog(sequence)`、`SkipCurrentLine()`、`HideDialog()` 等入口，并暴露完成事件。
   - 理由：满足“按需出现”的业务诉求，调用方无需了解内部协程细节。
   - 备选：调用方自行逐句驱动。缺点是重复逻辑多且一致性差。

## Risks / Trade-offs

- **[风险] 多次触发导致并发播放冲突** -> **Mitigation**：新会话启动前显式取消旧会话并重置 UI/音频状态。
- **[风险] 场景切换时引用丢失或音频未释放** -> **Mitigation**：在 `OnDisable/OnDestroy` 中统一清理协程与音频源状态。
- **[风险] 字速过慢影响节奏，过快丢失可读性** -> **Mitigation**：暴露字速配置并提供跳过当前句能力。
- **[权衡] 先实现基础样式，暂不覆盖复杂排版需求** -> **Mitigation**：预留文本样式和布局配置入口，后续迭代扩展。
