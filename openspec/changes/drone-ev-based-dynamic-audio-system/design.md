## Context

当前项目已经有无人机运动控制与场景逻辑，但声音系统尚未与运动状态形成结构化映射。用户提供了可用音频素材 `tsl audio`（电动车电机由加速到减速的一段连续录音），目标是在不重写现有控制系统的前提下，建立可参数化的动态无人机声音框架。

该变更横跨音频素材处理、运行时状态判定、控制系统集成三个模块，且需要平衡听感连续性与实现复杂度，因此需要先固化技术方案再进入编码阶段。

## Goals / Non-Goals

**Goals:**
- 让无人机在加速、减速、匀速、转弯、抬升、下降时具有可感知且连续的差异化音效。
- 基于 `tsl audio` 提取并构建可复用的音频层，而不是单一音频硬切换。
- 与现有无人机控制参数解耦对接，使用统一输入接口驱动声音系统。
- 支持后续调音（阈值、权重、过渡时间、滤波）而无需频繁改代码。

**Non-Goals:**
- 不在本次变更内实现完整 DCC 音频后期制作工具链（仅定义项目内可落地流程）。
- 不重构现有无人机飞控逻辑，仅消费其输出状态。
- 不引入多人联机同步音频策略。

## Decisions

1. 状态驱动 + 参数混合的双层模型  
   - 决策：采用“离散状态判定（加减速/转弯/升降）+ 连续参数调制（pitch/volume/LPF）”混合架构。  
   - 原因：仅用离散状态会造成切换突兀，仅用连续映射则难表达动作语义；双层组合可兼顾表达力和稳定性。  
   - 备选：纯状态机硬切，或纯 RTPC 曲线；前者听感差，后者调参成本高。

2. 以 `tsl audio` 做音色基底并拆分为功能层  
   - 决策：将源音频拆为至少 3 类层：基础持续层（匀速底噪）、动态冲量层（加减速感）、姿态修饰层（转向/升降变化）。  
   - 原因：无人机动作存在并发关系（例如转弯+加速），分层能进行叠加而非互斥替换。  
   - 备选：每种动作一条独立完整片段；但会导致素材爆炸且难维护。

3. 控制系统通过标准化 Telemetry 输入音频系统  
   - 决策：定义统一输入结构，至少包含 `forwardSpeed`, `acceleration`, `verticalSpeed`, `turnRate`, `throttle`。  
   - 原因：降低音频系统对具体控制器实现细节的耦合。  
   - 备选：音频脚本直接读取控制器私有字段；可快做但不可复用。

4. 过渡平滑必须内建防抖机制  
   - 决策：状态触发使用阈值 + 滞回（hysteresis）+ 最小保持时间，混合参数使用 damp/smoothing。  
   - 原因：飞行参数抖动会导致频繁切换，必须在系统层做抖动抑制。  
   - 备选：只靠音频淡入淡出，无法从源头稳定状态。

## Risks / Trade-offs

- [Risk] 单条 `tsl audio` 的频谱与无人机目标音色不完全匹配  
  → Mitigation: 在导入流程中保留 EQ/滤波与 pitch 区间配置，并允许后续替换同结构素材。

- [Risk] 状态阈值在不同场景速度尺度下不通用  
  → Mitigation: 所有阈值外置配置，按场景或机型做 preset。

- [Risk] 多音层并发带来响度堆叠和失真  
  → Mitigation: 设计总线限幅策略、分层最大增益限制与优先级 ducking。

- [Risk] 与现有控制脚本的更新频率不一致导致听感延迟  
  → Mitigation: 明确采样时序，统一在固定更新节点计算并缓存 telemetry。

## Migration Plan

1. 新增音频素材与配置资产（先不接线），完成导入参数和循环点标定。  
2. 新增运行时音频驱动脚本与 telemetry 接口，挂载到无人机预制体。  
3. 将当前控制系统参数接入 telemetry，启用状态判定和混合。  
4. 在测试场景逐项验证六类动作声音，再进行阈值与混合参数微调。  
5. 若出现明显回归，保留开关可回退到旧单轨音效方案。

## Open Questions

- `tsl audio` 是否为单声道/双声道、采样率与长度如何，是否需要先标准化处理？
- 当前无人机控制系统中可稳定读取的“转弯强度”参数是偏航输入还是角速度？
- 是否需要按第一/第三人称视角提供不同混音 preset？
